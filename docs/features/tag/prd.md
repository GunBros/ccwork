# 태그 기능 PRD

## 개요

노트 앱에 태그 기능을 추가한다. 사용자는 노트별로 태그를 붙이고 제거할 수 있으며, 노트 목록과 상세 화면에서 태그를 확인할 수 있다.

태그는 노트의 속성(`tags: string[]`)으로 저장되며, 별도 태그 컬렉션은 관리하지 않는다. 태그 추가·삭제는 로컬 상태에서만 즉시 반영되고, 기존 저장 버튼 흐름을 통해 서버에 반영된다.

---

## 사용자 스토리

### US-01. 태그 추가

> 노트를 작성하는 사용자로서,
> 노트에 태그를 추가하고 싶다.
> 그래야 노트를 분류하고 나중에 구분하기 쉽기 때문이다.

**수용 기준**

- NoteEditor에서 태그 입력창에 텍스트를 입력하고 Enter 또는 쉼표(,)를 누르면 태그가 확정된다.
- 쉼표 문자는 태그에 포함되지 않는다.
- 입력값 앞뒤 공백은 자동으로 제거된다.
- trim 후 빈 문자열이면 태그로 추가하지 않는다.
- 20자를 초과하는 입력은 무시한다.
- 같은 노트 안에 동일한 태그가 이미 있으면 추가하지 않는다(대소문자 구분).

### US-02. 태그 삭제

> 노트를 편집하는 사용자로서,
> 잘못 붙인 태그를 제거하고 싶다.
> 그래야 태그를 정확하게 유지할 수 있기 때문이다.

**수용 기준**

- 태그 뱃지의 `×` 버튼을 클릭하면 해당 태그가 즉시 제거된다.
- 삭제는 로컬 상태에서만 반영되며, 저장 버튼을 눌러야 서버에 반영된다.

### US-03. NoteEditor에서 태그 확인

> 노트를 편집하는 사용자로서,
> 현재 노트에 붙어 있는 태그를 한눈에 보고 싶다.
> 그래야 어떤 태그가 붙어 있는지 확인하면서 편집할 수 있기 때문이다.

**수용 기준**

- NoteEditor에 태그 목록과 입력창이 인라인으로 함께 표시된다.
- 형식: `[react ×] [hooks ×] [____________]`

### US-04. NoteList에서 태그 확인

> 노트 목록을 보는 사용자로서,
> 각 노트에 어떤 태그가 달려 있는지 목록에서 바로 보고 싶다.
> 그래야 노트를 열지 않고도 내용을 가늠할 수 있기 때문이다.

**수용 기준**

- NoteList의 각 노트 항목 아래에 태그가 뱃지 형태로 표시된다.
- 뱃지는 클릭 기능이 없다.

### US-05. 태그 포함 저장

> 노트를 편집하는 사용자로서,
> 태그 변경 사항을 기존 저장 버튼 한 번으로 저장하고 싶다.
> 그래야 제목·내용 저장과 흐름이 일관되기 때문이다.

**수용 기준**

- 저장 버튼 클릭 시 `updateNote({ ...note, tags })` API가 호출된다.
- Context에서 낙관적 업데이트로 로컬 목록이 즉시 갱신된다.

## 기술 결정 (ADR)

> 아키텍처 안 3(useTagEditor 커스텀 훅 + TagInput 분리) 채택을 기반으로 작성한다.
> 채택 이유: React 순수 API(useState, 커스텀 훅)를 활용한 합성 방식의 컴포넌트를 선호하며,
> 비즈니스 로직을 커스텀 훅으로 캡슐화하면 컴포넌트가 얇아지고 renderHook 기반 단위 테스트가 용이하다.

### ADR-01. 태그 저장 위치

- **Context**: 태그 데이터를 어디에 저장할지 결정해야 한다. 현재 JSON Server로 REST API를 모킹 중이며, `db.json`의 `notes` 배열이 유일한 데이터 저장소다. 태그를 노트 문서 안에 내장하거나 별도 컬렉션으로 분리하는 방식이 가능하다.

- **Decision**: `Note` 인터페이스에 `tags: string[]` 필드를 추가하고 노트 문서 내부에 인라인으로 저장한다. 별도 `/tags` 컬렉션은 만들지 않는다.

- **Alternatives**:
  - `/tags` 컬렉션 분리 (`{ id, name, noteId }` 관계형): 전역 태그 집계·검색에 유리하나 태그 조회마다 join이 필요하고 엔드포인트가 늘어남
  - localStorage 캐시: 서버 없이 빠르게 읽을 수 있으나 다기기 동기화 불가

- **Consequences**:
  - (+) `updateNote`가 이미 `Partial<Note>`를 수용하므로 API 레이어에 추가 엔드포인트가 불필요
  - (+) 노트 CRUD와 태그 CRUD가 단일 API 호출로 처리됨
  - (-) 전역 태그 목록 조회·집계 기능이 필요해지면 모든 노트를 순회해야 함 (현재 Out of Scope)

---

### ADR-02. 태그 입력 트리거 방식

- **Context**: 사용자가 입력창에 태그를 타이핑하다가 어느 시점에 태그를 "확정"할지 결정해야 한다. 트리거 선택은 UX 직관성과 허용 문자 범위에 영향을 미친다.

- **Decision**: **Enter** 키 또는 **쉼표(,)** 입력 시 태그 확정. 쉼표 문자 자체는 태그 값에 포함하지 않는다(`raw.replace(',', '').trim()`).

- **Alternatives**:
  - 스페이스바: 짧은 입력 흐름에 자연스러우나 공백 포함 태그 입력이 불가
  - 탭 키: 포커스 이동 기본 동작과 충돌하여 접근성 저해
  - blur(포커스 해제) 시 확정: 의도치 않은 확정이 발생할 수 있음
  - 별도 "추가" 버튼: 명시적이나 키보드 흐름이 끊김

- **Consequences**:
  - (+) GitHub·Notion 등 주류 태그 입력 UX와 동일하여 학습 비용 없음
  - (+) `useTagEditor` 훅의 `addTag` 함수에서 트리거 로직을 독립적으로 단위 테스트 가능
  - (-) 쉼표를 태그 텍스트로 사용할 수 없음

---

### ADR-03. 중복 검사 기준

- **Context**: 같은 노트 안에서 동일 태그를 중복 추가하는 것을 막아야 한다. 대소문자(`React` vs `react`)를 동일하게 볼지 구분할지 기준을 정해야 한다. 유효성 검사 로직은 `useTagEditor` 훅 내부에 캡슐화된다.

- **Decision**: **대소문자 구분(case-sensitive)** 비교. `tags.includes(value)` 단순 검사로 정규화 없이 구현한다.

- **Alternatives**:
  - 소문자 정규화(`toLowerCase()`) 후 비교: `React`와 `react`를 동일 취급하여 실수 방지
  - `Intl.Collator` 로케일 인식 비교: 한글 등 다국어 대소문자 처리에 정확하나 복잡도 증가

- **Consequences**:
  - (+) 구현이 한 줄(`tags.includes(value)`)로 단순하고 훅 단위 테스트가 쉬움
  - (+) 사용자가 의도적으로 `React`, `react`를 별개 태그로 사용 가능
  - (-) 대소문자만 다른 태그를 실수로 중복 추가할 수 있으나, 이는 UX 개선 영역으로 향후 검토

---

### ADR-04. 저장 시점 (즉시 저장 vs 명시적 저장)

- **Context**: 태그를 추가·삭제할 때마다 즉시 API를 호출하거나, 기존 저장 버튼 흐름에 통합할 수 있다. 저장 시점의 선택은 API 호출 빈도·사용자 경험·기존 패턴 일관성에 영향을 미친다.

- **Decision**: 태그 변경은 `useTagEditor` 훅의 로컬 state만 변경한다. 저장 버튼 클릭 시 `title`·`content`와 함께 `updateNote(id, { title, content, tags })`로 **단일 API 호출**로 반영한다.

- **Alternatives**:
  - 태그 추가·삭제마다 즉시 `updateNote` 호출(auto-save): 저장 버튼이 필요 없으나 API 호출이 빈번해지고 낙관적 업데이트 로직이 복잡해짐
  - 디바운스 후 자동 저장: 과다 호출을 줄이나 저장 타이밍이 불명확하고 사용자가 저장 여부를 인지하기 어려움

- **Consequences**:
  - (+) 기존 NoteEditor의 `title`/`content` 저장 패턴과 완전히 동일. 흐름 일관성 유지
  - (+) 여러 태그를 추가·삭제해도 API 호출은 저장 버튼 클릭 1회
  - (+) `useTagEditor`가 순수한 로컬 상태 관리만 담당하므로 훅 테스트 시 API 모킹 불필요
  - (-) 저장 버튼을 누르지 않고 다른 노트로 이동하면 태그 변경이 손실됨 (이탈 경고는 Out of Scope)

## Out of Scope

### 기능 측면

| 항목               | 설명                                                                            |
| ------------------ | ------------------------------------------------------------------------------- |
| 태그 기반 필터링   | 태그를 클릭하거나 선택해서 해당 태그가 붙은 노트만 보는 기능은 포함하지 않는다. |
| 태그 기반 검색     | 태그를 키워드로 노트를 검색하는 기능은 포함하지 않는다.                         |
| 전역 태그 관리     | 앱 전체에서 사용된 태그 목록을 별도로 조회·편집하는 화면은 없다.                |
| 태그 자동완성      | 기존에 사용한 태그를 입력 중에 제안하는 자동완성 기능은 포함하지 않는다.        |
| 태그 이름 변경     | 특정 태그를 일괄적으로 다른 이름으로 변경하는 기능은 포함하지 않는다.           |
| 태그 일괄 삭제     | 여러 노트에서 특정 태그를 한 번에 제거하는 기능은 포함하지 않는다.              |
| 태그 색상 지정     | 태그마다 색상을 지정하거나 카테고리를 나누는 기능은 포함하지 않는다.            |
| 태그 순서 변경     | 태그 뱃지를 드래그하거나 순서를 바꾸는 기능은 포함하지 않는다.                  |
| 태그 클릭 인터랙션 | NoteList의 태그 뱃지는 정보 표시 목적이며 클릭 이벤트를 처리하지 않는다.        |

### 데이터 측면

| 항목                | 설명                                                                             |
| ------------------- | -------------------------------------------------------------------------------- |
| 별도 tags 컬렉션    | `db.json`에 `tags` 최상위 컬렉션을 만들지 않는다. 태그는 노트 문서에만 존재한다. |
| 태그 사용 빈도 집계 | 어떤 태그가 몇 번 사용됐는지 집계·저장하지 않는다.                               |
| 태그 메타데이터     | 태그 설명, 생성일, 수정일 등 태그 자체에 대한 메타데이터는 관리하지 않는다.      |

### 유효성 검사 측면

| 항목            | 설명                                                                               |
| --------------- | ---------------------------------------------------------------------------------- |
| 허용 문자 제한  | 특수문자, 한글, 공백(trim 후 내부 공백) 등 문자 종류를 제한하지 않는다.            |
| 최소 길이 제한  | 태그 최소 글자 수 규칙은 없다(단, trim 후 빈 문자열만 제외).                       |
| 개수 제한       | 한 노트에 붙일 수 있는 태그 수에 상한을 두지 않는다.                               |
| 대소문자 정규화 | `React`와 `react`를 동일 태그로 취급하는 정규화는 하지 않는다(대소문자 구분 유지). |

### UX 측면

| 항목                     | 설명                                                                                    |
| ------------------------ | --------------------------------------------------------------------------------------- |
| 태그 입력 중 실시간 저장 | 태그를 추가·삭제할 때마다 즉시 API를 호출하지 않는다. 저장 버튼을 눌러야 반영된다.      |
| 저장 전 변경 경고        | 태그를 변경하고 저장하지 않은 채 다른 노트로 이동할 때 경고 다이얼로그를 띄우지 않는다. |
| 모바일 최적화            | 모바일 터치 환경을 별도로 고려하지 않는다.                                              |

## 용어 정의

| 용어            | 정의                                                                                  |
| --------------- | ------------------------------------------------------------------------------------- |
| 태그 (Tag)      | 노트에 부여하는 짧은 문자열 레이블. 노트 분류 목적으로 사용된다.                      |
| 태그 뱃지       | UI에서 태그를 표시하는 시각 요소. 태그 텍스트를 둘러싼 작은 칩(chip) 형태.            |
| 인라인 입력     | 태그 목록과 입력창이 별도 모달 없이 같은 행에 나란히 표시되는 UI 패턴.                |
| 태그 확정       | 입력창에서 Enter 또는 쉼표를 눌러 입력 중인 텍스트를 태그로 추가하는 동작.            |
| 로컬 상태       | 서버에 저장되지 않은, 컴포넌트 내부 `useState`로 관리되는 임시 상태.                  |
| 낙관적 업데이트 | API 응답을 기다리지 않고 로컬 상태를 먼저 갱신한 뒤 서버 결과로 동기화하는 패턴.      |
| trim            | 문자열 앞뒤의 공백 문자를 제거하는 처리. JavaScript의 `String.prototype.trim()` 사용. |
| 대소문자 구분   | 중복 검사 시 `React`와 `react`를 서로 다른 태그로 취급하는 기준.                      |
