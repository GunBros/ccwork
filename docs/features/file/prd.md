# 파일 첨부 기능 PRD

## 개요

노트 앱에 파일 첨부 기능을 추가한다. 사용자는 노트에 이미지·텍스트 파일을 첨부하고, 첨부된 파일을 확인·다운로드·삭제할 수 있다.

파일은 Base64로 인코딩되어 노트의 속성(`files: AttachedFile[]`)으로 `db.json`에 저장된다. 파일당 최대 2MB, 노트당 최대 2개까지 첨부할 수 있으며, 허용 확장자는 이미지(jpg, jpeg, png, gif, webp)와 텍스트(txt, md)로 제한된다.

---

## 사용자 스토리

### US-01. 파일 추가

> 노트를 작성하는 사용자로서,
> 노트에 파일을 첨부하고 싶다.
> 그래야 관련 자료를 노트와 함께 보관할 수 있기 때문이다.

**수용 기준**

- NoteEditor 하단의 파일 첨부 버튼을 클릭하면 파일 선택 다이얼로그가 열린다.
- 허용 확장자(jpg, jpeg, png, gif, webp, txt, md)만 선택할 수 있다.
- 허용되지 않은 파일 타입 선택 시 에러 메시지를 표시하고 차단한다.
- 2MB 초과 파일 선택 시 에러 메시지를 표시하고 차단한다.
- 노트당 2개 초과 첨부 시도 시 에러 메시지를 표시하고 차단한다.
- 동일 파일명이 이미 첨부되어 있으면 에러 메시지를 표시하고 차단한다.

### US-02. 파일 삭제

> 노트를 편집하는 사용자로서,
> 잘못 첨부한 파일을 제거하고 싶다.
> 그래야 첨부 파일을 정확하게 유지할 수 있기 때문이다.

**수용 기준**

- 각 파일 항목의 삭제 버튼을 클릭하면 해당 파일이 목록에서 제거된다.
- 삭제는 로컬 상태에서 즉시 반영된다.

### US-03. 첨부 파일 목록 확인

> 노트를 편집하는 사용자로서,
> 현재 노트에 첨부된 파일을 한눈에 보고 싶다.
> 그래야 어떤 파일이 첨부되어 있는지 확인하면서 편집할 수 있기 때문이다.

**수용 기준**

- NoteEditor 하단에 첨부 파일 목록이 표시된다.
- 이미지 파일은 썸네일 형태로 표시된다.
- 텍스트 파일은 파일 이름만 표시된다.

### US-04. 파일 다운로드

> 노트를 확인하는 사용자로서,
> 첨부된 파일을 PC에 다운로드하고 싶다.
> 그래야 첨부 파일을 로컬에서 활용할 수 있기 때문이다.

**수용 기준**

- 파일 항목을 클릭하면 원본 파일명으로 PC에 다운로드된다.

### US-05. 노트 삭제 시 첨부 파일 처리

> 노트를 삭제하는 사용자로서,
> 노트 삭제 시 첨부 파일도 함께 삭제되기를 기대한다.
> 그래야 불필요한 데이터가 남지 않기 때문이다.

**수용 기준**

- 노트 삭제 시 첨부 파일도 함께 삭제된다.
- 별도 경고 없이 즉시 삭제한다.

## 기술 결정 (ADR)

> 아키텍처 안 3(useFileAttachment 커스텀 훅 + FileList 분리) 채택을 기반으로 작성한다.
> 채택 이유: 태그 기능에서 확립한 커스텀 훅(useTagEditor) + Presentational 컴포넌트(TagInput) 분리 패턴을
> 파일 첨부에도 동일하게 적용하여 프로젝트 전체의 일관성을 유지한다.
> 비즈니스 로직(유효성 검사, Base64 변환)을 훅으로 캡슐화하면 renderHook 기반 단위 테스트가 용이하다.

### ADR-01. 파일 저장 방식

**Context** — 첨부 파일의 바이너리 데이터를 어디에, 어떤 형태로 저장할지 결정해야 한다. 현재 백엔드는 JSON Server(`db.json`)이며 파일시스템 접근이나 별도 스토리지 API가 없다.

**Decision** — 파일을 Base64로 인코딩하여 `Note.files: AttachedFile[]` 필드에 인라인 저장한다. `AttachedFile`은 `{ id, name, type, size, data }` 구조이며, `data`에 Base64 문자열을 담는다. 별도 `/files` 컬렉션이나 외부 저장소는 사용하지 않는다.

**Alternatives**

- `/files` 별도 컬렉션 분리 (`{ id, noteId, data }`): 노트 목록 조회 시 파일 데이터를 불필요하게 로드하지 않아도 되지만, 저장·삭제 시 두 엔드포인트를 호출해야 하고 트랜잭션 보장이 없음
- localStorage/IndexedDB 클라이언트 저장: 서버 부하 없이 빠르지만 다기기 동기화 불가, 브라우저 용량 제한에 의존
- 외부 파일 저장소(S3 등): 대용량에 적합하지만 JSON Server 목 환경에서 구현 불가

**Consequences**

- (+) `updateNote`가 이미 `Partial<Note>`를 수용하므로 API 레이어에 추가 엔드포인트 불필요
- (+) 노트 CRUD와 파일 CRUD가 단일 API 호출로 처리됨
- (-) Base64 인코딩으로 원본 대비 ~33% 용량 증가. 2MB 파일이 ~2.7MB JSON 문자열이 됨
- (-) 노트 목록 조회(`fetchNotes`) 시 모든 파일 데이터가 함께 로드됨 (현재 규모에서는 허용 가능)

### ADR-02. 유효성 검사 처리 위치

**Context** — 파일 크기(2MB), 개수(2개), 허용 확장자, 동일 파일명 중복 등 유효성 검사 로직을 어디에 배치할지 결정해야 한다. 검사 로직의 위치는 테스트 용이성과 재사용성에 직접 영향을 미친다.

**Decision** — 모든 유효성 검사를 `useFileAttachment` 커스텀 훅 내부에 캡슐화한다. 훅의 `addFile` 함수가 검사를 수행하고, 실패 시 에러 메시지를 `error` state로 반환한다. `FileList` 컴포넌트는 에러를 표시만 하고 검사 로직을 갖지 않는다.

**Alternatives**

- 컴포넌트(NoteEditor) 내부에서 직접 검사: 별도 훅 없이 단순하지만, 검사 로직이 렌더 사이클에 묶여 격리 테스트 불가
- API 레이어(`src/api/notes.ts`)에서 검사: 서버 측 검증과 유사하지만, 사용자에게 피드백이 늦고 불필요한 Base64 변환이 먼저 발생
- 별도 유틸 함수로 추출(`src/utils/fileValidation.ts`): 순수 함수로 테스트 용이하지만, 상태(현재 파일 개수)에 접근하려면 매번 인자로 전달해야 하고 훅과 이중 구조가 됨

**Consequences**

- (+) `renderHook`으로 모든 검사 규칙을 컴포넌트 렌더 없이 단위 테스트 가능
- (+) 태그 기능의 `useTagEditor` 패턴과 동일한 구조 → 프로젝트 내 일관성 유지
- (+) 에러 state를 훅이 관리하므로 컴포넌트는 `error`를 렌더만 하면 됨
- (-) 훅 내부에 `FileReader` 비동기 로직(Base64 변환)이 포함되어, 태그 훅보다 테스트 설정이 복잡해짐

### ADR-03. 파일 식별 및 중복 검사 기준

**Context** — 첨부 파일을 삭제할 때 어떤 값으로 식별할지, 동일 파일 중복 첨부를 어떤 기준으로 판단할지 결정해야 한다. 파일에는 고유 식별자가 자연적으로 존재하지 않는다.

**Decision** — 삭제 식별은 `AttachedFile.id`(클라이언트에서 생성하는 고유 ID)를 사용한다. 중복 검사는 **파일명(`name`) 기준 대소문자 구분 비교**로 수행한다(`files.some(f => f.name === newFile.name)`).

**Alternatives**

- 파일명 + 크기 조합으로 중복 검사: 동명이파일(같은 이름, 다른 내용)을 허용할 수 있지만, 사용자 혼란 유발
- Base64 data 해시 비교: 내용 기반으로 정확하지만 해시 계산 비용이 크고, 이름만 다른 동일 파일도 차단됨
- id만으로 중복 검사: 매번 새 id가 생성되므로 중복 검사 자체가 불가

**Consequences**

- (+) 구현이 단순 — `files.some(f => f.name === name)` 한 줄
- (+) 사용자 관점에서 직관적 — "같은 이름의 파일이 이미 있습니다"
- (-) 이름만 바꾸면 동일 내용의 파일을 중복 첨부할 수 있음 (허용 가능한 트레이드오프)

### ADR-04. 저장 시점 (즉시 저장 vs 명시적 저장)

**Context** — 파일을 첨부·삭제할 때마다 즉시 API를 호출하거나, 기존 저장 버튼 흐름에 통합할 수 있다. 파일은 태그와 달리 Base64 변환이 필요하고 데이터 크기가 크므로 저장 전략이 UX와 성능에 영향을 미친다.

**Decision** — 파일 첨부·삭제는 `useFileAttachment` 훅의 로컬 state만 변경한다. 저장 버튼 클릭 시 `title`·`content`·`tags`와 함께 `updateNote(id, { title, content, tags, files })` **단일 API 호출**로 반영한다.

**Alternatives**

- 파일 첨부 즉시 `updateNote` 호출(auto-save): 저장 버튼이 필요 없으나 2MB Base64 데이터를 매 첨부마다 전송하게 되어 비효율적. 삭제 후 되돌리기 불가
- 파일만 별도 엔드포인트로 즉시 저장, 메타데이터는 노트와 함께: 파일 업로드 실패를 독립 처리할 수 있으나 JSON Server에 커스텀 엔드포인트를 만들어야 하고 저장 흐름이 이원화됨

**Consequences**

- (+) 기존 NoteEditor의 title/content/tags 저장 패턴과 완전히 동일 — 흐름 일관성 유지
- (+) 여러 파일을 첨부·삭제해도 API 호출은 저장 버튼 클릭 1회
- (+) `useFileAttachment`가 순수한 로컬 상태 관리만 담당하므로 훅 테스트 시 API 모킹 불필요
- (-) 저장 버튼을 누르지 않고 다른 노트로 이동하면 첨부 변경이 손실됨 (이탈 경고는 Out of Scope)
- (-) 파일 크기가 크므로 저장 시 네트워크 지연이 태그보다 체감될 수 있음

## Out of Scope

### 기능 측면

| 항목                  | 설명                                                         |
| --------------------- | ------------------------------------------------------------ |
| 파일 미리보기         | 첨부 파일을 모달이나 별도 뷰어에서 전체 크기로 미리보는 기능 |
| 드래그 앤 드롭 업로드 | 파일을 드래그하여 에디터에 놓는 방식의 업로드                |
| 파일 이름 변경        | 첨부 후 파일 이름을 수정하는 기능                            |
| 파일 정렬·순서 변경   | 첨부 파일의 표시 순서를 변경하는 기능                        |
| 파일 교체             | 기존 첨부 파일을 다른 파일로 교체하는 기능                   |
| NoteList 첨부 표시    | 노트 목록에서 첨부 파일 유무를 아이콘으로 표시하는 기능      |

### 데이터 측면

| 항목                 | 설명                                                          |
| -------------------- | ------------------------------------------------------------- |
| 외부 파일 저장소     | S3, 로컬 파일시스템 등 외부 저장소를 사용하지 않는다          |
| 파일 메타데이터 확장 | 업로드 일시, 업로더 정보 등 추가 메타데이터는 관리하지 않는다 |

| 항목               | 설명                                                                   |
| ------------------ | ---------------------------------------------------------------------- |

| 모바일 최적화      | 모바일 터치 환경을 별도로 고려하지 않는다                              |

## 용어 정의

| 용어             | 정의                                                                                   |
| ---------------- | -------------------------------------------------------------------------------------- |
| 첨부 파일        | 노트에 추가된 파일. `AttachedFile` 타입으로 표현된다.                                  |
| Base64 인코딩    | 바이너리 데이터를 ASCII 문자열로 변환하는 방식. 파일을 JSON에 저장하기 위해 사용한다.  |
| MIME type        | 파일의 미디어 종류를 나타내는 표준 식별자. `image/png`, `text/plain` 등.               |
| 썸네일           | 이미지 파일의 축소된 미리보기. 파일 목록에서 이미지를 구분하기 위해 표시한다.          |
| 허용 확장자      | 첨부 가능한 파일 형식. 이미지(jpg, jpeg, png, gif, webp)와 텍스트(txt, md)로 제한된다. |
| 파일당 최대 크기 | 단일 파일의 용량 상한. 2MB로 제한된다.                                                 |
| 노트당 최대 개수 | 한 노트에 첨부할 수 있는 파일 수 상한. 2개로 제한된다.                                 |
